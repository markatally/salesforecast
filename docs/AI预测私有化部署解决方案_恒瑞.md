# 1. 系统架构与流程

## 1.1 设计原则

- **数据不出内网**：客户数据全程存储于客户自有云服务器，倍通方无法访问原始数据。
- **最小化依赖**：部署环境标准化，减少客户侧运维负担。
- **可维护性**：预留远程运维通道，支持模型迭代与版本升级。

## 1.2 系统架构图

| 数据源 | 模型计算专属服务器 | 输出层 |
|---|---|---|
| 客户数据；客户原始数据库 | 客户提供服务器；建模输入数据；倍通预测引擎 | 服务器/指定环境；预测结果；日志文件 |
| 数据不出内网 | 客户数据传输；倍通模型部署 | 模型输出存储 |

说明：数据流向为单向（从数据库到内网服务器），预测结果在服务器本地生成，倍通方仅通过交付预测引擎和定期维护的形式完成服务。

## 1.3 核心交互流程

1. 恒瑞方准备数据，按约定格式放置于指定位置。
2. 定期触发预测引擎，读取输入数据，并产生预测结果。
3. 预测结果以约定格式写入输出目录，并记录运行日志。
4. 倍通方通过日志报告或 VPN 通道进行远程性能监控与模型迭代。

# 2. 客户提供数据规范

## 2.1 数据提供形式

文件格式：数据仓库、CSV（UTF-8 编码）或 Excel（`.xlsx`），推荐数据仓库。

### 数据仓库形式

若选择数据仓库形式，则在数据库中维护如下表：

| 表名 | 说明 |
|---|---|
| `ods_sales_flow` | 产品流向数据（必须，可脱敏处理） |
| `ods_org_master` | 机构主数据（可选，可脱敏处理） |
| `ods_business_drivers` | 其他业务驱动因素（可选，如资源投入、管理指标等） |
| `dws_forecast_result` | 预测结果输出目录（系统自动写入） |
| `dws_forecast_model_log` | 运行日志目录（系统自动写入） |

### CSV 或 Excel 文件形式

若选择 CSV 或 Excel 格式，则在服务器指定根目录下维护以下子文件夹：

| 目录路径 | 说明 |
|---|---|
| `/data/input/sales/` | 产品流向数据（必须，可脱敏处理） |
| `/data/input/org/` | 机构主数据（可选，可脱敏处理） |
| `/data/input/drivers/` | 其他业务驱动因素（可选，如资源投入、管理指标等） |
| `/data/output/` | 预测结果输出目录（系统自动写入） |
| `/data/logs/` | 运行日志目录（系统自动写入） |

## 2.2 数据质量要求

- **时间连续性**：至少提供 24 个月及以上的数据。
- **缺失值**：核心字段（年月、销量等）不允许为空。
- **关联主键**：各数据文件中保持一致且具有唯一性。
- **异常值处理**：退货、盘点调整等异常记录请在提交前进行标记或剔除。
- **脱敏说明**：如经过脱敏，仍需说明字段含义。

# 3. 硬件与软件环境要求

## 3.1 服务器硬件配置

| 配置项 | 最低要求 | 推荐配置 | 备注 |
|---|---|---|---|
| CPU | 8 核 / 3.0 GHz | 16 核 / 3.5 GHz+ | 支持 AVX2 指令集 |
| 内存（RAM） | 32 GB | 64 GB | 特征矩阵、模型训练 |
| 存储（SSD） | 20 GB | 50 GB SSD | 数据存储、模型文件、日志 |
| GPU | 无需 | 可选 NVIDIA A6000 | 仅当使用深度学习辅助模型时需要 |
| 网络 | 内网互通 | 内网 + 可选 VPN | VPN 用于远程运维通道 |

## 3.2 操作系统与软件依赖

- **操作系统**：Ubuntu 20.04 LTS / 22.04 LTS（推荐）或 CentOS 7+。
- **Python 运行时**：Python 3.9–3.11（建议使用 Conda 隔离环境）。
- **核心依赖**：Python 库明细由倍通提供。
- **Docker（推荐）**：倍通可提供容器化镜像，确保环境一致性。

调度服务可选：

|  | 无调度 | Linux Crontab | Apache Airflow |
|---|---|---|---|
| 适合场景 | 月度执行频率低、IT 资源有限 | 月度固定时间自动执行，希望减少人工介入 | 多产品线并行、多步骤依赖，需要完整监控平台 |
| 执行方式 | 每月手动运行命令 | 系统在指定时间自动触发 | 通过 Web UI 管理和触发 DAG 任务流 |
| 失败通知 | 无自动通知，需人工检查输出目录 | 可配置邮件告警，成功/失败均发通知 | 内置告警、重试、Web UI 日志可视化 |
| 运维要求 | 无 | 低 | 高 |

## 3.3 网络安全要求

- 服务器需位于客户内部网络或私有云，禁止公网直接访问数据目录。
- 若需要倍通方远程运维，建议开通 VPN 通道，仅允许特定 IP 白名单访问。
- 倍通预测引擎不主动发起任何外网请求，License 验证支持离线模式。

# 4. 倍通交付内容

## 4.1 交付物清单

| 交付物 | 描述 |
|---|---|
| 模型文件 | 包含所有模型文件及完整运行环境的容器镜像，部署后可直接运行。 |
| License 文件 | 绑定客户服务器 CPU ID / 机器 UUID，防止拷贝到其他机器运行；License 设置有效期（如每年更新），到期后自动失效，保障持续合作。 |
| 部署手册 | 覆盖 Docker 安装与启动步骤、数据目录挂载配置、定时任务设置、模型切换操作说明等内容。 |
| 版本说明文档 | 记录当前交付版本的新增功能、模块变更日志、问题修复等内容。 |

## 4.2 发布与交付流程

1. 倍通方在安全计算环境中完成模型开发与测试。
2. 倍通方将模型及依赖库封装为 Docker 镜像。
3. 倍通方生成针对特定服务器的 License 文件，绑定机器 ID 和有效期。
4. 倍通方通过安全渠道将 Docker 镜像和 License 文件交付客户。
5. 恒瑞方按部署手册将文件放置到指定目录，运行初始化脚本完成安装。
6. 倍通方远程确认运行状态，完成交付验收。

## 4.3 版本管理

- 每次代码迭代生成新版本号（如 `v1.2.0`），保留历史版本回滚能力。
- 建议客户保留最近两个版本备份，以便异常时快速回滚。

## 5.4 初始建模流程

1. **数据探索**：在安全计算环境中进行数据探查与质量评估。
2. **特征工程**：基于数据探索性分析结果设计模型特征。
3. **模型选型与结构确定**：通过比较多个模型性能确定最终选用模型。
4. **超参数调优**：通过交叉验证方式对超参数调优。
5. **回测验证与交付**：通过双方可接受的方式对模型精度进行评估，经双方确认后进入模型部署阶段。

## 5.5 定期运行机制

### Docker 容器化

将模型文件和 Python 依赖封装在单一 Docker 镜像中。客户无需考虑 Python 环境和依赖版本，只需在服务器上运行一条命令启动，或者由调度工具自动触发。

### 数据质检模块

数据质检内置于容器中，每次运行自动执行。质检逻辑分为三个级别：

- **阻断级（Blocker）**：发现问题则流程终止，写入错误日志并发送告警。检查项包括当月数据文件是否存在、核心字段是否有空值、时间格式是否符合规范等。
- **警告级（Warning）**：发现问题后流程继续运行，但在报告中标记。检查项包括当月销量相比过去 6 个月均值异常（超过 3 倍标准差，可能是录入错误）、新覆盖机构等。
- **信息级（Info）**：记录数据基本统计摘要，方便人工审阅。

### 自动调参模块

预测引擎支持定期重新搜索超参数（如每季度更新），并对新模型做回测验证。通过后替换线上版本。调参完成后新模型不会自动上线，而是先写入指定目录（例如 `models/candidate/`），生成候选模型评估报告，经人工审阅确认后提升为正式版本。

# 6. 模型性能监控与更新

## 6.1 运维模式说明

- 本系统所依赖的数据，包括销售流向、销售指标、市场投入等数据，客户需要按约定格式进行准备，并及时更新。
- 本系统稳定运行期间，客户端本地部署引擎可自动执行全流程。
- 倍通方仅参与模型精度审查等日常运维，并在出现数据结构调整、模型精度衰减等异常情况时进行干预。

## 6.2 模型性能监控机制

1. 每月预测完成后，系统自动生成误差监控报告（CSV + 可视化图表）。
2. 倍通每月审阅报告，判断是否触发模型复查。
3. 倍通分析误差原因（数据漂移、外部冲击、特征失效等），制定调整方案。
4. 执行调整（特征更新 / 重训练 / 重新部署），并记录变更日志。

## 6.3 模型更新触发场景

- 药品政策重大变化（如医保目录大幅调整）。
- 新竞品上市或重磅产品退市导致市场结构改变。
- 季节性规律出现显著漂移（如突发事件影响）。
- 客户产品组合扩大（新增 SKU 超过原有规模 20% 以上）。
- 其他内外部冲击。

## 6.4 模型版本与迭代策略

- 每次重训练生成新模型版本，旧版本保留 3 个月备用。
- 受到重大外部冲击后，建议 1 个月内完成紧急迭代。
- 每次迭代后需在历史数据集上回测，精度改善方可替换线上版本。
